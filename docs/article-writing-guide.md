# 記事執筆ガイド(コンテンツ制作部向け)

コンテンツ制作部が記事(`src/content/articles/*.md`)を執筆する際の実務ルールです。
事業としての執筆ガイドライン骨子(ハンド選定基準・情報源の扱い方・著作権チェックリスト等)は
会社側リポジトリの企画書を正とします。**このドキュメントを読む前に必ず以下を読んでください。**

- [`cloude会社/projects/poker-media/README.md`](../../../../cloude会社/projects/poker-media/README.md) の
  「記事執筆ガイドライン骨子」「記事の型(標準構成)」節

このファイルは、その内容をこのリポジトリの実装(フロントマタースキーマ・コンポーネント)に
どうマッピングするかを示す技術的な手引きです。

## 1. 記事の型(標準構成)とこのリポジトリでの書き方

企画書の「記事の型」7セクションは、以下のように対応します。

| # | 記事の型 | このリポジトリでの書き方 |
|---|---|---|
| 1 | 導入 | 本文冒頭、見出しなしのリード文 |
| 2 | ハンド経過 | **推奨**: フロントマターの `handProgression` に構造化して入力する(`HandProgression` コンポーネントがストリート別の表として自動出力する。下記5-3参照)。この場合、本文に `## ハンド経過` の見出し・番号付きリストは重複して書かない。用語解説記事などハンド進行そのものを扱わない記事では省略してよい(旧方式: 本文 `## ハンド経過` の見出しの下に番号付きリストで書くと丸数字ステップ風に表示される。`handProgression` を使わない場合の代替手段として引き続き利用可) |
| 3 | プレイヤーの思考プロセスの解説 | 本文 `## プレイヤーの思考プロセス` の見出し |
| 4 | 判断の評価 | 本文 `## 判断の評価` の見出し |
| 5 | エクイティ目安 | 本文 `## エクイティ目安` の見出し。厳密な数値ではなくレンジ表現(例:「およそ6〜7割」)で書く。表にすると見やすい |
| 6 | 学びのポイント | 本文 `## 学びのポイント` の見出し |
| 7 | 出典明記 | **本文中に手書きしない。** フロントマターの `sources` フィールドに構造化して入力する(下記2参照)。記事下部に `SourceList` コンポーネントが自動出力する |

## 2. フロントマターの必須・任意フィールド

`src/content.config.ts` のスキーマ定義を正とします。特に以下はこのメディア特有のフィールドです。

- `editorialNote`(必須): 編集方針メモ。「配信・記事の解説を手がかりに、プレイヤーの思考プロセス・評価を独自の言葉で再構成した」旨を明記する。テンプレ的な一文で済ませず、その記事固有の情報源・観点に触れること
- `tournament`(任意推奨): 大会名・イベント名。例: `"WSOP 2026 メインイベント"`
- `tournamentDate`(任意。**必須にしない**): 大会自体の開催日付。`pubDate`(このメディアでの記事の公開日)とは別物であることに注意(取材・執筆はハンド発生から日を置いて行われることが多く、両者は一致するとは限らない)。**取材元(`sources`)や本文に開催時期の記載が実際にある場合のみ入力する。判明していない記事ではこのフィールドごと省略してよい。** 過去にこのメディアでは大会の開催時期を無理に特定しようとして事実誤認を起こした経緯がある(`triton-one-jeju-wang-martirosian-fold-kings.md` の `editorialNote` 参照。候補となる開催回が複数あり一次ソースだけでは絞り込めなかったため、開催年月・回次の特定自体を取り下げた実例)。詳しい書き方・根拠の示し方は下記5-2Aを参照
- `players`(任意推奨): 登場プレイヤー名の配列。タグとして記事上部に表示される
- `sources`(出典。可能な限り入力): `label`(配信名・記事名)と`url`のペアの配列。記事下部に自動出力される(著作権チェックリストの「出典明記」を機械的に担保する)
- `featuredHands`(任意): 記事タイトル周りに大きく表示する「見どころハンド」。詳細は下記5-4参照
- `affiliateProducts`: Amazonアソシエイト等の**物販**アフィリのみ。オンラインカジノ/オンラインポーカー(実マネー)のアフィリエイトは会社の方針として非採用(企画書「収益化・法務リスク」節参照)。設置しない場合は空配列`[]`のままでよい
- `relatedServices`: 自社サービス(`poker-tourney-log`、`fukuoka-poker-navi`)への送客。広告ではなく内部リンクのため「広告」表記は付かない
- `draft`: **デフォルトは`true`**。品質管理部の事実確認・著作権チェック、レビュー部のGO判定が終わるまでは`true`のままにする。`true`のままでも`npm run dev`のローカル開発サーバーでは表示される(レイアウト確認用)。`npm run build`(本番/プレビューデプロイ用ビルド)では自動的に除外される

## 3. 著作権チェックリスト(執筆時に必ず自己確認)

企画書の「著作権チェックリスト」をそのまま適用します。

- [ ] 配信・記事の文章をそのまま書き写していないか(書き起こし・無断転載になっていないか)
- [ ] 記事の主題が「解説者の発言の要約」になっていないか(プレイヤーの思考プロセス・評価として独自に組み立て直されているか)
- [ ] 動画を使う場合、ダウンロード・再アップロードではなく公式YouTubeの埋め込み機能(iframe)を改変せず使用しているか(下記4参照)
- [ ] 出典(配信名・チャンネル名・記事名)を`sources`フィールドに明記しているか
- [ ] 引用箇所がある場合、必要最小限にとどまっているか(記事の主体は独自解説であること)

## 4. Markdown記法ルール(セマンティック規約)

カスタムコンポーネントタグの埋め込みは不要です。以下のプレーンなMarkdown記法で書けば、
デザイン側(`src/pages/articles/[...slug].astro` の `.prose` スタイル)が自動的に見栄えを整えます。

### ハンド経過(ステップ)

番号付きリスト(`1. 2. 3.`)で書くと、大きな丸数字のステップ風デザインになります。

```markdown
1. プリフロップ: Aがオープンレイズ、Bが3ベット
2. フロップ: K♠K♥7♦。Aがベット、Bがコール
```

### 学びのポイント・注意

引用ブロック(`>`)の先頭を「💡」で始めるとポイント、「⚠️」で始めると注意のコールアウト風になります。

```markdown
> 💡 ベットサイズの小ささは必ずしも弱さを意味しない。
> ⚠️ エクイティは目安レベルであり、厳密なソルバー計算ではない。
```

### エクイティ目安(比較表)

Markdownテーブルで書くと、ヘッダー強調+ストライプ模様の見やすいテーブルになります。

```markdown
| プレイヤー | フロップ時点のおよその勝率感 |
| --- | --- |
| A | およそ6〜7割 |
```

### 配信動画の埋め込み

YouTube公式の埋め込みコード(`<iframe src="https://www.youtube.com/embed/...">`)を
改変せず、そのまま本文に貼り付けてください。レスポンシブな16:9の枠に自動整形されます。
動画のダウンロード・切り出し・再アップロードは著作権上のリスクが高いため行わないでください。

## 5. ビジュアル部品(カードバッジ / ハンド進行表 / トーナメント情報欄)

2026-07に追加した部品群。文字だけの解説から、カード・ストリート経過・トーナメント状況をひと目で把握できる見せ方に強化するためのもの。
実装(コンポーネント本体)は `src/components/` を参照。ここでは執筆側の使い方のみ説明する。

### 5-1. カードバッジ(ランク+スートの色分け表示)

**新しい記法を覚える必要はない。** これまで通り本文に `A♠` `K♠K♥7♦` のようにランク+スート記号を続けて書くだけでよい。
ビルド時に `rehypeCardBadges`(`src/lib/rehype-card-badges.ts`。`astro.config.mjs` に登録済み)が自動的に検出し、
ハート/ダイヤは赤、スペード/クラブは黒(濃色)の角丸カードバッジに変換する。見出し直下・本文中・テーブルのセル内、どこに書いても効く。

```markdown
プリフロップで**A♠A♦**を握るマターンに対し、**K♣K♦**のトレドが3ベット。
```

上記はそのまま書くだけで `A♠` `A♦` `K♣` `K♦` がそれぞれ個別のカードバッジとして表示される。

- 対象はスート記号(♠♥♦♣)付きの表記のみ。`10♣` のように2桁ランクもそのまま書いてよい(`T♣` と書いても同じ見た目になる)
- インラインコード・コードブロック内は変換対象外
- 表構成のコンポーネント(`handProgression` の `board` 等)側では、代わりに `CardBadge.astro` コンポーネントが同じロジックでカード文字列(例: `"K♠"` `"10♦"`)をパースして表示する。執筆側がこのコンポーネントを直接呼び出す必要はなく、フロントマターに文字列でカードを書けばよい

### 5-2. トーナメント情報欄(`tableContext`)

参加人数・ブラインドレベル・残り人数・平均スタックと、本人(主役)・相手それぞれのスタック/ポジションを、記事冒頭にコンパクトな情報バーとして自動表示する。
**ハンド解説記事でのみ設定する**(用語解説記事など、特定のハンドの前提を持たない記事ではフィールドごと省略する。省略すれば何も表示されない)。

フロントマターに以下のように追加する(すべて任意。情報源に記載がない項目は無理に埋めず省略してよい)。

```yaml
tableContext:
  entrants: "8,663人"                # 参加人数
  blindLevel: "1,000 / 2,500"        # ブラインドレベル(SB/BB。アンティがあれば "(アンティ 2,500)" のように併記)
  playersRemaining: "3,000人以上"     # 残り人数
  avgStack: "230,000"                # 平均スタック
  heroName: "デルミロ・トレド"          # 本人(主役)の名前
  heroStack: "230,000"               # 本人のスタック
  heroPosition: "カットオフ"           # 本人のポジション(ボタン/カットオフ/SB/BB等)
  villainName: "アーノー・マターン"     # 相手の名前
  villainStack: "350,000"            # 相手のスタック
  villainPosition: "アーリーポジション" # 相手のポジション
```

数値は文字列(`z.string()`)で持つため、カンマ区切りや「約」「〜以上」などの表現をそのまま書いてよい。3人以上が絡むハンドでは、記事の主役(判断を評価する対象)を `hero`、その判断に最も関わる相手を `villain` として1名ずつ選んで書く(残りの登場人物は本文中で通常通り説明すればよい)。

### 5-2A. 大会の開催日付(`tournamentDate`)

社長要望(2026-07)により追加。記事本文の見出し直下・トップページの記事カード/大会グループ見出し付近に、その**大会自体が開催された時期**を表示する任意フィールド(`tableContext`ではなくフロントマター直下の項目。`tournament`のすぐ下に書く)。

```yaml
tournament: "WSOP 2026 メインイベント"
tournamentDate: "2026年7月"   # 大会自体の開催日付(任意。判明している場合のみ)
```

**絶対に守ること: 判明していない場合は省略する。憶測・推測で埋めない。** このフィールドを必須にしていないのはそのためです。`pubDate`(このメディアでの記事の公開日)は「編集部がいつ記事を書いたか」であり、大会が実際に開催された時期とは別物です。特に、取材元の報道記事は大会からしばらく経ってから配信されることもあれば(過去に発生した実例: `triton-jeju-ii-foxen-ketola-11-million-pot.md` は `pubDate: 2026-07-23` ですが、出典のPokerNews記事は2025年9月に公開されたものであり、`tournamentDate` は `"2025年9月"` です)、逆にライブ速報のように大会期間中とほぼ同時に配信されることもあり、一律の法則はありません。**`pubDate` の年月をそのまま `tournamentDate` に転記することは絶対にしないでください。**

書き方・判断基準:

- 記事本文または出典(`sources`)に大会の開催日・開催月が明記されている場合は、その記載をそのまま使う
- 明記が無くても、出典として採用したニュース記事のURL自体に公開年月が含まれている場合(例: PokerNewsの記事URLは `https://www.pokernews.com/news/2026/07/...` のように `/news/YYYY/MM/` の形式で公開年月を含むことが多い)、それを月単位の根拠として使ってよい。この場合は `editorialNote` に「出典URLの公開年月(YYYY年M月)を根拠にtournamentDateを設定した」旨を一言明記すること(サンプル記事のeditorialNote参照)
- 日付を1つに特定できない場合(候補が複数ある、期間しか分からない等)は、無理に1点に絞らず期間表記("2026年7月14日〜7月16日")や月単位("2026年7月")など、実際に判明している粒度でとどめる
- どうやっても大会の時期の手がかりが無い場合は、フィールドごと省略する。「不明」「未定」等のダミー文字列を入れない(省略すれば表示・ソートの両方で自動的に「不明」として扱われる)
- 西暦4桁を含めること。トップページのソート(`src/pages/index.astro`)は文字列内の西暦4桁(・月・日)を正規表現で読み取って新しい順に並べる仕組みのため、西暦4桁が無いとそのグループは「開催日不明」扱いになり、ソート上は末尾側(不明グループの中)に回る
- 同じ`tournament`名を共有する記事(同じ大会シリーズの複数ハンド記事)は、原則同じ`tournamentDate`を使う。トップページのグループ見出しには、そのグループ内で最初に見つかった`tournamentDate`が代表値として表示される

### 5-3. ハンド進行表(`handProgression`)

プリフロップ→フロップ→ターン→リバーの各ストリートについて、その時点のボードカード・ポットサイズ・実際のアクションを表形式で自動表示する。
**このフィールドを設定した場合、本文(Markdown)側に `## ハンド経過` の見出し・番号付きリストを重複して書かない**(この構造化データがその節の役割を担うため)。

```yaml
handProgression:
  - street: "プリフロップ"
    board: []                        # プリフロップはボードが無いため空配列
    pot: "9,000"                     # そのストリート終了時点のポットサイズ
    actions:
      - player: "アーノー・マターン"
        action: "オープンレイズ"
        amount: "5,500"
        hand: ["A♠", "A♦"]           # このプレイヤーのハンド(任意)。選手名の横にカードバッジで表示される
      - player: "デルミロ・トレド"
        action: "3ベット"
        amount: "20,000"
        hand: ["K♣", "K♦"]
      - player: "デルミロ・トレド"
        action: "フォールド"          # amount は省略可(フォールド等、額を伴わないアクション)。hand も同様に省略可
  - street: "フロップ"
    board: ["9♠", "6♥", "2♣"]        # そのストリート時点までに開いたボードカード
    pot: "82,000"
    actions:
      - player: "アーノー・マターン"
        action: "ベット"
        amount: "40,000"
  - street: "ターン"
    board: ["9♠", "6♥", "2♣", "7♠"]
    pot: "162,000"
    actions: []                      # チェック等で新たな金額の動きが無い場合は空配列でよい
  - street: "リバー"
    board: ["9♠", "6♥", "2♣", "7♠", "J♦"]
    pot: "930,000"
    actions:
      - player: "アーノー・マターン"
        action: "A♠A♦がK♠K♥に勝利しポット獲得"
```

- `street` は自由記述(基本は「プリフロップ」「フロップ」「ターン」「リバー」の4つ)
- `board` の各カードは5-1と同じ表記(`"K♠"` `"10♦"` 等)。`CardBadge` コンポーネントが自動でパースする
- オールイン後のランアウトなど、そのストリートで新たなベットが発生しない場合は `actions: []` のままでよい(ボードカードとポットだけが表示される)
- ハンドがプリフロップ中に完結した(オールイン合戦の末フォールドで決着した)場合、その後のフロップ/ターン/リバーは「ランアウトのため金額変動なし」である旨を `pot` の文字列に注記するとよい(サンプル記事 `wsop-2026-main-event-aces-vs-kings-vs-kings.md` 参照)
- `actions[].hand`(任意): そのアクション時点でのプレイヤーのハンド(ホールカード)。設定すると選手名の横に `(K♣J♥)` のようにカードバッジで表示される。**主にプリフロップのアクションで使う**(そのプレイヤーが最初にアクションした行にだけ書けば十分。同じプレイヤーの2回目以降のアクションでは繰り返し書く必要はない)。書き方は5-1・`board` と同じ、カード表記を1枚ずつ配列で書く(例: `["K♣", "J♥"]`)。フォールドしたプレイヤーなど、情報源にハンドの記載が無い場合は省略してよい

### 5-4. 主役ハンドバッジ(`featuredHands`)

記事のH1タイトル直下、およびトップページの記事一覧カードに、その記事の「見どころハンド」を大きめのカードバッジで目立たせて表示する。文字だけのタイトルよりも「どんなハンドの対決か」がひと目で伝わり、クリック率の向上を狙った部品(`FeaturedHands` コンポーネント)。

```yaml
# ヘッズアップ(1対1)の場合
featuredHands: ["K♣J♥", "8♦6♦"]

# 3-wayポットなど、3人以上が絡むハンドの場合(要素数は可変)
featuredHands: ["A♠A♦", "K♣K♦", "K♠K♥"]
```

- 配列の各要素は5-1と同じ「ランク+スート記号を連結しただけ」の書き方(例: `"K♣J♥"`)。新しい記法を覚える必要はない
- 要素数は対戦人数に応じて自由(ヘッズアップなら2つ、3-wayポットなら3つ)。表示側は要素の間を自動的に「vs」で区切る
- 用語解説記事など、特定のハンドの前提を持たない記事では**フィールドごと省略する**(省略・空配列であれば何も表示されない)
- `handProgression` を設定している記事では、通常そのハンドのプリフロップの各プレイヤーの `hand`(上記)と同じカードを入れることになる

## 6. 執筆前の確認事項

- ハンド選定基準(盛り上がったハンドの定義)・品質ゲート(思考プロセスを読み解ける情報源があるか)は
  企画部が選定段階で確認済みの前提で執筆に着手する
- 出典が複数ある場合は、それぞれの見解を統合した上で記事側の独自解釈・評価を書く
- サンプル記事 `src/content/articles/wsop-2026-main-event-aces-vs-kings-vs-kings.md` は
  開発部が投入したレイアウト確認用のプレースホルダーです。`tableContext` / `handProgression` の
  書き方(フロントマターの構造)は参考にしてよいですが、「参考値・要確認」と注記した数値は
  情報源に無い仮の値です。実際の執筆時は数値の参考にはせず、必ず出典を精査した上でゼロから執筆してください
