# 記事執筆ガイド(コンテンツ制作部向け)

コンテンツ制作部が記事(`src/content/articles/*.md`)を執筆する際の実務ルールです。
事業としての執筆ガイドライン骨子(ハンド選定基準・情報源の扱い方・著作権チェックリスト等)は
会社側リポジトリの企画書を正とします。**このドキュメントを読む前に必ず以下を読んでください。**

- [`cloude会社/projects/poker-media/README.md`](../../../../cloude会社/projects/poker-media/README.md) の
  「記事執筆ガイドライン骨子」「記事の型(標準構成)」節

このファイルは、その内容をこのリポジトリの実装(フロントマタースキーマ・コンポーネント)に
どうマッピングするかを示す技術的な手引きです。

## 1. 記事の型(標準構成)とこのリポジトリでの書き方

企画書の「記事の型」7セクションは、以下のように対応します。

| # | 記事の型 | このリポジトリでの書き方 |
|---|---|---|
| 1 | 導入 | 本文冒頭、見出しなしのリード文 |
| 2 | ハンド経過 | 本文 `## ハンド経過` の見出し。プリフロップ→フロップ→ターン→リバーを番号付きリストで整理すると丸数字ステップ風に表示される |
| 3 | プレイヤーの思考プロセスの解説 | 本文 `## プレイヤーの思考プロセス` の見出し |
| 4 | 判断の評価 | 本文 `## 判断の評価` の見出し |
| 5 | エクイティ目安 | 本文 `## エクイティ目安` の見出し。厳密な数値ではなくレンジ表現(例:「およそ6〜7割」)で書く。表にすると見やすい |
| 6 | 学びのポイント | 本文 `## 学びのポイント` の見出し |
| 7 | 出典明記 | **本文中に手書きしない。** フロントマターの `sources` フィールドに構造化して入力する(下記2参照)。記事下部に `SourceList` コンポーネントが自動出力する |

## 2. フロントマターの必須・任意フィールド

`src/content.config.ts` のスキーマ定義を正とします。特に以下はこのメディア特有のフィールドです。

- `editorialNote`(必須): 編集方針メモ。「配信・記事の解説を手がかりに、プレイヤーの思考プロセス・評価を独自の言葉で再構成した」旨を明記する。テンプレ的な一文で済ませず、その記事固有の情報源・観点に触れること
- `tournament`(任意推奨): 大会名・イベント名。例: `"WSOP 2026 メインイベント"`
- `players`(任意推奨): 登場プレイヤー名の配列。タグとして記事上部に表示される
- `sources`(出典。可能な限り入力): `label`(配信名・記事名)と`url`のペアの配列。記事下部に自動出力される(著作権チェックリストの「出典明記」を機械的に担保する)
- `affiliateProducts`: Amazonアソシエイト等の**物販**アフィリのみ。オンラインカジノ/オンラインポーカー(実マネー)のアフィリエイトは会社の方針として非採用(企画書「収益化・法務リスク」節参照)。設置しない場合は空配列`[]`のままでよい
- `relatedServices`: 自社サービス(`poker-tourney-log`、`fukuoka-poker-navi`)への送客。広告ではなく内部リンクのため「広告」表記は付かない
- `draft`: **デフォルトは`true`**。品質管理部の事実確認・著作権チェック、レビュー部のGO判定が終わるまでは`true`のままにする。`true`のままでも`npm run dev`のローカル開発サーバーでは表示される(レイアウト確認用)。`npm run build`(本番/プレビューデプロイ用ビルド)では自動的に除外される

## 3. 著作権チェックリスト(執筆時に必ず自己確認)

企画書の「著作権チェックリスト」をそのまま適用します。

- [ ] 配信・記事の文章をそのまま書き写していないか(書き起こし・無断転載になっていないか)
- [ ] 記事の主題が「解説者の発言の要約」になっていないか(プレイヤーの思考プロセス・評価として独自に組み立て直されているか)
- [ ] 動画を使う場合、ダウンロード・再アップロードではなく公式YouTubeの埋め込み機能(iframe)を改変せず使用しているか(下記4参照)
- [ ] 出典(配信名・チャンネル名・記事名)を`sources`フィールドに明記しているか
- [ ] 引用箇所がある場合、必要最小限にとどまっているか(記事の主体は独自解説であること)

## 4. Markdown記法ルール(セマンティック規約)

カスタムコンポーネントタグの埋め込みは不要です。以下のプレーンなMarkdown記法で書けば、
デザイン側(`src/pages/articles/[...slug].astro` の `.prose` スタイル)が自動的に見栄えを整えます。

### ハンド経過(ステップ)

番号付きリスト(`1. 2. 3.`)で書くと、大きな丸数字のステップ風デザインになります。

```markdown
1. プリフロップ: Aがオープンレイズ、Bが3ベット
2. フロップ: K♠K♥7♦。Aがベット、Bがコール
```

### 学びのポイント・注意

引用ブロック(`>`)の先頭を「💡」で始めるとポイント、「⚠️」で始めると注意のコールアウト風になります。

```markdown
> 💡 ベットサイズの小ささは必ずしも弱さを意味しない。
> ⚠️ エクイティは目安レベルであり、厳密なソルバー計算ではない。
```

### エクイティ目安(比較表)

Markdownテーブルで書くと、ヘッダー強調+ストライプ模様の見やすいテーブルになります。

```markdown
| プレイヤー | フロップ時点のおよその勝率感 |
| --- | --- |
| A | およそ6〜7割 |
```

### 配信動画の埋め込み

YouTube公式の埋め込みコード(`<iframe src="https://www.youtube.com/embed/...">`)を
改変せず、そのまま本文に貼り付けてください。レスポンシブな16:9の枠に自動整形されます。
動画のダウンロード・切り出し・再アップロードは著作権上のリスクが高いため行わないでください。

## 5. 執筆前の確認事項

- ハンド選定基準(盛り上がったハンドの定義)・品質ゲート(思考プロセスを読み解ける情報源があるか)は
  企画部が選定段階で確認済みの前提で執筆に着手する
- 出典が複数ある場合は、それぞれの見解を統合した上で記事側の独自解釈・評価を書く
- サンプル記事 `src/content/articles/wsop-2026-main-event-aces-vs-kings-vs-kings.md` は
  開発部が投入したレイアウト確認用のプレースホルダーです。実際の執筆時の参考にはせず、
  必ず出典を精査した上でゼロから執筆してください
